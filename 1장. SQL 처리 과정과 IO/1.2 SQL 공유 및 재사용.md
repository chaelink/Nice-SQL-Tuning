#### 1.2.1 소프트 파싱 vs 하드 파싱

라이브러리 캐시
- SQL 최적화 전 과정을 거쳐 생성한 내부 프로시저를 반복 재사용할 수 있도록 캐싱해 두는 메모리 공간
- SGA 구성요소


소프트 파싱
- SQL을 캐시에서 찾아 곧바로 실행단계로 넘어가는 것

하드 파싱
- 캐시에서 찾는 데 실패해 최적화 및 로우 소스 생성 단계까지 모두 거치는 것

<BR>

바인드 변수
- sql문에서 동적으로 변하는 데이터를 직접 기술하는 대신, ?나 : 같이 자리 표시자로 기술하여 재사용성을 높이는 변수

바인드 변수의 중요성
- 하드파싱은 최초 한 번만 일어나고, 캐싱된 SQL문을 재사용 가능하게 함

sql은 이름이 없다. 따라서, sql 텍스트가 이름 역할을 한다.
sql 조건절을 `price = 00` 이런 형식으로 설정하면 내용은 같지만, 매번 새로운 sql이 생성되어
실행 시 마다 새 프로시저를 만들어 라이브러리 캐시에 저장하게 되고 이는 곧 과도한 하드파싱으로 cpu 사용량이 급격하게 증가하게 돈다. => 매우 큰 비효율

따라서, 바인드 변수를 사용하여 price 값을 파라미터로 받는 프로시저 하나를 공유하며 재사용되도록 sql을 작성하면
하드파싱은 최초 1회만 발생하고, 이후부터는 라이브러리 캐시에 캐싱된 프로시저를 재사용하게 된다.
